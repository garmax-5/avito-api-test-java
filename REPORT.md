##### **Отчет о тестировании**



###### **Общая информация:**

1\. В связи с отсутствием технической документации и формального описания бизнес-логики сервиса, тест-кейсы на валидацию бизнес-ограничений параметров запросов (диапазоны значений, взаимосвязи полей, бизнес-правила) не разрабатывались. 

2\. В рамках тестирования была выполнена проверка корректности обработки обязательных полей, типов данных, структуры запросов и ответов, а также обработки невалидных входных данных.

3. В данном файле собрал проверки, которые основывались на предположениях и анализе swagger.yml, application\_collection и API ответов.



###### **Обнаруженные баги/проблемы:**



1\. В postman\_collection обнаружено несогласованность в названиях полей в теле запроса и ответа (sellerId и sellerID) для некоторых запроса используется название sellerID для других запросов sellerId.



2\. Если в POST-запросе /api/1/item поле sellerId передать как тип <string> то возвращается тело запроса вида:

{

    "result": {

        "message": "",

        "messages": {}

    },

    "status": "не передано тело объявления"

}



3\. Если в POST-запросе /api/1/item для поля likes задать значение 0 то возвращается тело запроса вида:

{

    "result": {

        "message": "поле likes обязательно",

        "messages": {}

    },

    "status": "400"

}

Аналогично для полей viewCount и contacts.



4\. Если в POST-запросе /api/1/item поле likes передать как тип <string> то возвращается тело запроса вида:

{

    "result": {

        "message": "",

        "messages": {}

    },

    "status": "не передано тело объявления"

}

Аналогично для полей viewCount и contacts.



5\. Если в POST-запросе /api/1/item для поля likes задать отрицательное значение то запрос выполнится и объявление сохранится. Аналогично для полей viewCount и contacts.



6\. Если в POST-запросе /api/1/item для поля price задать отрицательное значение, то запрос выполнится и объявление сохранится, однако при значении 0 то возвращается тело запроса:

{

    "result": {

        "message": "поле price обязательно",

        "messages": {}

    },

    "status": "400"

}



7\. Если в POST-запросе /api/1/item поле price передать как тип <string>, то возвращается тело запроса вида:

{

    "result": {

        "message": "",

        "messages": {}

    },

    "status": "не передано тело объявления"

}



8\. Если в POST-запросе /api/1/item в поле name передать пустую строку (""), то возвращается тело запроса:

{

    "result": {

        "message": "поле name обязательно",

        "messages": {}

    },

    "status": "400"

}



9\. Если в POST-запросе /api/1/item отправить невалидный JSON то возвращается тело запроса:
{

    "result": {

        "message": "",

        "messages": {}

    },

    "status": "не передан объект - объявление"

}



10\. Если в GET-запросе /api/1/{sellerID}/item передать sellerID несуществующего продавца, то сервер вернет HTTP-status: 200 OK и пустой список. Предполагаю что должен возвращаться HTTP-status: 404 Not Found и тело ответа с информацией о том что продавца с таким идентификатором не существует.



11\. Если в GET-запросе /api/2/statistic/{id} передать невалидный id, то сервер вернет HTTP-status: 404 Not Found. Лучше возвращать HTTP-status: 400 Bad Request.



12\. Во время выполнения теста была обнаружена проблема с блокировкой: POST-запрос /api/1/item не возвращает созданный объект с сгенерированным идентификатором. Из-за этой проблемы тесты, зависящие от возвращаемого идентификатора, не могут быть успешно выполнены.
